Syntax

  • Use soft tabs with two spaces for Rails, or four spaces for PHP â€” they're the only way to guarantee code renders the same in any environment.
  • Use the Sass indented syntax.
  • Styles should be separated out in to appropriate files/folders, see the accompanying folder structure to get a good idea of where things should go.
  • Comma-separated values should include a space after each comma.
  • Make use of Sass color functions e.g. rgba(#000, 0.5) and lighten(#000, 10%).
  • Lowercase all HEX values.
  • Use shorthand HEX values where possible.
  • Quote attribute values in selectors, e.g. input[type="text"].
  • Avoid specifying units for zero values, e.g. margin: 0 instead of margin: 0px.
  • Use white space to separate logical blocks of code, e.g. a blank line between modifiers / elements.

Bourbon

We make use of the bourbon library, they have lots of useful mixins to save you lots of time, they also save you having to add vendor prefixes.

.my-block
  +position(absolute, 10px null null 20px)
  +size(200px 300px)
  +padding(20px)
  +transition(background-color 0.3s)

Sass - Minxins, Functions & Variables

Sass has a lot of handy features to help help you code, cleaner, smarter and quicker, we suggest you familiarise yourself with them and start using them.

Variables

  • Global variables should be stored in an appropriate file, e.g. fonts.sass, in the Settings folder.
  • Variables specific to a certain block / template should be stored at the top of the appropriate file.
  • Variable names should be lowercase and hyphenated.

Mixins/Functions

  • We have a set of Rawnet, pre-defined mixins/functions.
  • When creating your own, mixins/functions should be stored in their own file in the tools folder under their appropriate sub-folder, e.g. tools/mixins/grid.sass.
  • Try not to get too obscure with your mixins/functions, they can help you save a lot of time, but if someone new comes to the project, they need to understand what everything does.
  • Study the accompanying folder structure to get a good idea of where things should go.

BEM

We follow the BEM (Block Element Modifier) coding standard, it helps make our CSS maintainable and scalable, while keeping specificty as low as possible.

.widget
  width: 200px
  background-color: #fff
  border: 1px solid #ccc

  &--modifier
    background-color: #ff6600
    .widget__element
      display: block

  &__element
    display: inline-block
    font-size: 20px
    color: #ff6600

    &--modifier
      color: #fff

Declaration Order

Related property declarations should be grouped together following the order:

  1. Positioning
  2. Box Model
  3. Visual
  4. Typographic

Positioning comes first because it can remove an element from the normal flow of the document and override box model related styles. The box model comes next as it dictates a component's dimensions and placement.

Everything else takes place inside the component or without impacting the previous two sections, and thus they come last.

.declaration-order
  // Positioning
  position: absolute
  top: 0
  right: 0
  bottom: 0
  left: 0
  z-index: 1

  // Box model
  display: block
  float: right
  width: 100px
  height: 100px

  // Visual
  background-color: #fff
  border: 1px solid #ccc
  border-radius: 3px

  // Typography
  font: normal 13px arial
  line-height: 1.5
  text-align: center
  color: #ddd

  // Misc
  opacity: 1

Media Query Placement

Place media queries as close to their relevant rule sets whenever possible. Don't bundle them all in a separate stylesheet, doing so only makes it easier for folks to miss them in the future.

Shorthand Notation

Strive to limit use of shorthand declarations to instances where you must explicitly set all the available values. Common overused shorthand properties include:

  • padding
  • margin
  • font
  • background
  • border
  • border-radius

Often times we don't need to set all the values a shorthand property represents. For example, HTML headings only set top and bottom margin, so when necessary, only override those two values. Excessive use of shorthand properties often leads to sloppier code with unnecessary overrides and unintended side effects.

You can use bourbon's padding/margin mixins to avoid setting values unnecessarily, e.g. +padding(10px null 0 null).

Class Names

Firstly, always style using classes, not IDs.

  • Keep class names lowercase and hyphenated.
  • Avoid abbreviations, it may make sense to you, but it might not to someone else.
  • Keep class names as short and succinct as possible.
  • Use meaningful names; use structural or purposeful names over presentational.
